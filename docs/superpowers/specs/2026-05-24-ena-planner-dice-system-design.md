# EnaPlanner Optional Dice System Design

**Date:** 2026-05-24
**Status:** Approved for implementation planning

## Objective

Add an optional, editable dice-rules prompt module to LittleWhiteBox EnaPlanner. When enabled, the planner receives a fixed dice pool for the current turn. If the planner cannot produce usable injected planning content, the final response model receives the same already-resolved dice rules and pool, without any re-roll.

## Existing Architecture

EnaPlanner configuration is stored through `EnaPlannerStorage` in `LittleWhiteBox_EnaPlanner.json`. Its prompt designer already models editable blocks as `promptBlocks` and actual send ordering as `moduleChain`. Prompt templates save copies of both structures.

`buildPlannerMessages(rawUserInput)` currently visits `moduleChain` in order and renders each editable prompt block with `renderTemplateAll(...)`. That renderer runs SillyTavern macro substitution through `substituteParamsExtended(...)`.

`runPlanningOnce()` builds the request messages, calls the planning API, and filters its response. `doInterceptAndPlanThenSend()` then appends the filtered planning result to the original player input and permits the normal final-response send.

Both current SillyTavern macro engines support the user-specified dice syntax `{{roll:1d20}}`.

## Requirements

1. Add an independent dice-system switch in EnaPlanner basic settings.
2. Add dice rules as an editable module in the unified module chain; its send order can be adjusted.
3. Save edited dice text and its module-chain position through existing configuration and prompt-template persistence.
4. During normal planning, the planner model receives resolved dice rules and uses the generated results when producing planning instructions.
5. If the filtered planning result is empty, append the same resolved dice rules to the input sent to the final-response model.
6. If the planning API returns empty content, throws, or times out, append the same resolved dice rules to the final-response model instead of stopping the send.
7. Never re-render or re-roll dice for the final-response fallback within the same turn.
8. Do not alter existing non-dice module behavior when the dice switch is disabled.

## Chosen Approach

Implement the dice rules as one protected editable `promptBlock` with a fixed internal ID. It participates in the existing `promptBlocks` and `moduleChain` model rather than introducing a third module type.

The dice block:

- Is always present after settings and template normalization.
- Is fixed to `system` role.
- Can have its content edited and can be moved within the unified module chain.
- Cannot be deleted or individually disabled.
- Is activated only by the independent `diceSystem.enabled` setting.

This preserves the existing module and template model while ensuring that there is exactly one authoritative dice prompt for each configuration.

The editable text is the user-owned rule/pool content. Small planner-facing and fallback-facing operational wrappers are generated by runtime code and are not exposed as editable prompt text; they only tell the receiving model how to apply the same resolved rules in that path.

## Configuration And Migration

Add `diceSystem.enabled`, defaulting to `false`, to EnaPlanner configuration. It is a global setting and is not switched when a user selects a prompt template.

Add a dice helper module responsible for:

- The fixed dice block ID and default content.
- Normalizing `diceSystem` settings.
- Ensuring exactly one protected dice block and one matching chain entry exist.
- Building the per-turn resolved dice content and final-response fallback content.

During `ensureSettings()`:

- If a current configuration lacks the protected dice block or its chain entry, insert it once using default content.
- If it already exists, retain the user's edited content and chain position.
- Normalize its role and identity as protected system content.

During prompt template normalization:

- Add the missing protected block/entry to legacy or existing templates that lack it.
- Preserve stored dice content and order in templates that already have it.
- Template save/load includes the protected block and chain position through existing `promptBlocks` and `moduleChain` copies.

Restoring default prompt configuration resets the dice block content and its default position along with the other default modules, but does not change the global dice enable switch.

## Settings UI

In the basic settings page, add a dice-system selector with `关闭` and `开启` states. Its value maps only to `diceSystem.enabled`.

In the unified module-chain editor, render the protected dice block as a special editable prompt card:

- Display its fixed identity and `system` role.
- Display a note that activation is controlled in basic settings.
- Provide content editing and move-up/move-down controls.
- Omit delete and module-level enable controls.

In preview/test help text, state that enabling dice resolves a new test dice pool for that preview or test invocation; it is not the pool used by a later formal send.

## Turn Data Flow

### Dice Disabled

When `diceSystem.enabled` is `false`, visiting the protected module produces no planner message and does not call macro rendering for its content. Existing planner behavior remains unchanged.

### Dice Enabled And Planning Succeeds

For a formal send:

1. `buildPlannerMessages()` gathers existing context and visits modules in configured chain order.
2. When it reaches the protected dice block, it calls `renderTemplateAll()` once for that block content. This resolves all `{{roll:1d20}}` values for this turn.
3. It adds a planner-facing `system` message at that exact chain position, containing the resolved rules and an instruction to use the fixed dice pool in planning output without re-rolling.
4. It retains a final-response fallback wrapper containing the same resolved rules in per-turn metadata.
5. Only after message construction completes does the planning API request begin.

If the filtered planner response is non-empty, the final response receives only the original player input plus the filtered planning result, as in current behavior. The full dice-rule block is not appended to the final-response input on this successful path.

### Empty Planning Result

If the planner response is empty or becomes empty after filtering, `doInterceptAndPlanThenSend()` appends the cached final-response fallback wrapper to the original player input. The wrapper tells the final-response model to apply the already-generated dice values directly.

The fallback uses the previously rendered string. It must not call `renderTemplateAll()` or `substituteParamsExtended()` again and must contain no unresolved `{{roll...}}` macros.

### Planner Error Or Timeout

The resolved dice context is created before `callPlanner()` begins. The runtime must retain that context outside the failing API call so the error branch can access it.

If the planning API throws or times out while the dice system is enabled and a resolved dice fallback exists:

- Record the planning failure in existing logging/error diagnostics.
- Treat the planner result as unavailable.
- Append the cached final-response fallback wrapper to the original input.
- Permit the normal final-response send.

No new dice context may be built inside this error path.

If the dice system is disabled, planning API errors retain current failure behavior because there is no enabled dice fallback.

### Final Response Failure

Once EnaPlanner has released the send and the final-response request has been submitted, EnaPlanner does not automatically send it again on downstream error or timeout. It cannot reliably tell whether a failed client request has already produced server-side output; retrying risks duplicate responses.

## Logging And Preview

Planner request logs continue to store actual request messages. When dice is enabled, those messages expose the resolved pool used for the planner request, which supports diagnosis of ordering and fairness.

Preview and planner-test actions construct independent test turns. They may resolve and display a dice pool for inspection, but they do not establish or consume the pool used by a future formal send.

## Anticipated Code Areas

- `modules/ena-planner/ena-planner-dice.js`: protected block constants, default prompt, migration helpers, per-turn resolution/fallback helpers.
- `modules/ena-planner/ena-planner-presets.js`: include the default protected dice block in default prompt content/templates.
- `modules/ena-planner/ena-planner.js`: settings normalization, module-chain generation, per-turn resolved context propagation, empty/error/timeout fallback path.
- `modules/ena-planner/ena-planner.html`: basic setting control, protected card behavior, preview/test explanatory text and persistence plumbing.
- `modules/ena-planner/tests/dice-system.test.js`: focused behavior coverage.
- `package.json`: add a dedicated EnaPlanner dice test script so the new focused suite has a stable verification command.

## Verification

Automated tests should prove:

- Dice settings default to off and normalize/save correctly.
- The protected block is added once to new and migrated configuration/templates.
- Existing edited dice content and order survive normalization and template save/load.
- The protected block cannot be represented as disabled/deleted through normalization.
- Disabled dice do not trigger macro rendering or modify existing planner messages.
- Enabled dice resolve module content exactly once per constructed formal turn.
- Planner and fallback wrappers contain identical resolved dice results.
- The fallback wrapper contains no remaining dice macro syntax.
- A non-empty filtered planner result does not append dice rules to final-response input.
- Empty/filtered-empty planner output appends the cached resolved fallback.
- Planner exceptions and timeouts append the same cached resolved fallback rather than generating a new pool.

Manual validation in the EnaPlanner UI should cover:

- Toggling and persisting the independent switch.
- Editing and saving dice content.
- Moving the protected module and verifying preview send order.
- Saving/loading templates and restoring defaults.
- Running normal, empty-result, and planner-error/timeout paths with dice enabled.
- Running existing modules with dice disabled to confirm no visible regression.

## Non-Goals

- Implementing DND character sheets, modifiers, HP calculation, or an internal rules engine.
- Persisting a dice pool between separate sends, previews, or tests.
- Retrying final-response failures after the send has already been released.
- Changing unrelated EnaPlanner modules or external SillyTavern macro semantics.
