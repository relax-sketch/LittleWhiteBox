# 剧情规划接入聊天记录管理器经验总结

日期：2026-05-05

## 背景

小白X的剧情规划器曾为聊天记录管理器（vectors-enhanced）单独维护一套向量知识库设置，包括召回数量、相似度阈值、任务选择、查询指令和注入模板。实测中，这会让规划器和聊天记录管理器的真实配置分叉：用户以为当前聊天管理器设置生效，但剧情规划实际走的是另一套参数。

## 本次调整

- 快捷开关暂时关闭“聊天记录管理器(软)”和“小白X(软)”两个入口，只保留 WestWorld 相关入口，避免跨扩展软开关在设置或 DOM 未加载时误切。
- 小白X剧情规划器不再保存独立的向量查询参数，只保留是否调用聊天记录管理器。
- 剧情规划器把玩家最新输入当作普通 query 传给 `window.VectorsEnhanced.queryForPrompt()`，查询任务、阈值、召回数量、查询指令、重排等全部使用聊天记录管理器现有设置。
- 聊天记录管理器的对外查询接口在 storage 未初始化、总开关关闭、查询开关关闭时返回空结果和 `stats.reason`，不再抛出异常打断调用方流程。

## 经验

1. 跨扩展集成不要复制对方的业务配置。调用方应传最小输入，让被调用方用自己的设置决策。
2. 公共 API 不应该因为“未准备好/关闭”这种可预期状态抛异常；返回空结果和明确原因更适合上层降级。
3. 软开关依赖被控扩展的设置对象和 DOM checkbox，适合作为扩展内部功能，不适合在第三方快捷菜单里长期维护。
4. 诊断入口必须走和真实调用相同的参数路径，否则排查结果会误导。
5. 查询失败不应该导致剧情规划失败。向量召回只是增强上下文，主流程应继续生成规划。

## 后续建议

- 聊天记录管理器可以继续收敛 `queryForPrompt()` 这类公共 API，明确哪些参数允许外部覆盖，哪些必须使用自身设置。
- 如果以后恢复快捷软开关，应优先走稳定的公开 API，而不是直接改其他扩展的内部设置字段或 DOM。
