# EnaPlanner 可选骰子系统：调查与设计计划

## 目标

在修改功能代码前，梳理 LittleWhiteBox 中 EnaPlanner 的模块链、模板保存、配置保存以及规划/正文兜底发送流程，形成经用户确认的骰子系统设计。

## 约束

- 用户确认设计前不修改功能代码。
- 骰子系统应可开关、可排序、可编辑保存，并在同轮规划与空回正文兜底间复用解析后的骰子结果。
- 现有模块行为不得受到影响。

## 已确认的产品选择

- 骰子模块默认内容使用用户提供的 `<合理性审查>` 与包含 10 个 `D20` 的 `<骰子池>` 文本。
- 骰子模块始终保留在统一模块链内，可编辑且可排序，但不可删除、不可通过模块卡片单独关闭；是否执行仅由基本设置中的独立总开关决定。
- 当规划过滤结果为空时，将本轮已经解析过宏的骰子规则文本追加到原始玩家输入后，再沿用现有发送流程交给正文模型；禁止重新执行骰子宏。
- 采用推荐实现路径：固定 ID 的受保护 `promptBlock` 进入现有 `promptBlocks/moduleChain`，由单轮上下文在模块遍历位置解析一次骰子宏并承担空回复用。
- 配置/链设计已确认：`diceSystem.enabled` 为全局开关；骰子块固定为 `system`、可编辑可排序但不可删除或单独关闭；模板携带块内容及位置但不携带总开关；旧配置/模板缺块时自动补入且不覆盖已有编辑或排序。
- 运行时失败策略已确认：规划消息构造先解析骰子模块，随后才调用规划 API；因此规划 API 空响应、抛错或超时时已有同轮骰池，均走已解析骰池的正文兜底，而不是恢复原输入并终止。
- 正文请求已经由放行发送提交后的失败不由 EnaPlanner 自动重发，以避免重复正文生成。
